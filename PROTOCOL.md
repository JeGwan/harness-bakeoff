# Fairness Protocol

각 회차(run)는 아래 규칙을 **반드시 동일하게** 따른다. 위반한 run은 matrix.md에 "⚠️ protocol violation" 표기.

## 1. 세션 격리

- **반드시 새 세션**에서 시작. 이전 세션의 컨텍스트가 남아 있으면 안 됨.
- Claude Code 기준: 새 창을 열거나 `/clear` 후 시작.
- 평가용 문서를 그동안 본 적이 있는 세션/계정은 회차 진행 금지.

## 2. 입력 컨텍스트 제한

하니스에 노출되는 파일은 **`spec/prompt.md` 한 문단뿐**.
다음은 **절대 입력에 포함하지 않는다:**

- `spec/notes.md` (정답지)
- `matrix.md`
- `runs/**` (다른 하니스 결과)
- `README.md`, `PROTOCOL.md`
- 이전 회차의 산출물

가장 안전한 방법: **prompt.md 한 문단을 손으로 복사해 빈 디렉토리/세션에 그대로 붙여넣기.**

## 3. 환경 동일화

- 동일 모델 (가능하면 동일 버전·날짜). 어쩔 수 없이 다르면 notes.md에 명시.
- steering / CLAUDE.md / AGENTS.md / .cursorrules 등 **하니스 기본 컨벤션 외 사전 컨텍스트 금지**.
  - 단, 하니스 자체가 요구하는 파일(예: Kiro의 `.kiro/steering/*`, BMAD의 역할 정의)은 그 하니스의 본질이므로 허용. notes.md에 그 내용을 기록.
- 동일 권한 모드 (자동 승인 OFF 권장 — 사람 개입 횟수를 정확히 카운트하기 위해).

## 4. 사람 응답 표준화

하니스가 결정을 사람에게 위임할 때, 다음 규칙으로 답한다:

- **기본 응답: "당신이 합리적으로 판단해 진행하세요."** (모호 지점을 누가 채우는지 측정하기 위함)
- **예외 1**: 하니스가 진행 자체를 막는 hard block (예: "이 결정 없이는 다음 단계 불가") → 가장 단순한 옵션 선택 후 notes.md에 기록.
- **예외 2**: 명백한 환경 정보 질문 (예: "OS는?", "Node 버전은?") → 사실대로 답.

답변 횟수와 내용 모두 `runs/<harness>/notes.md`에 기록.

## 5. 종료 조건

다음 중 하나로 종료:

1. 하니스가 "완료" 선언 + 첫 실행이 성공.
2. 90분 경과 또는 5회 이상의 막힘(stuck) 발생.
3. 사람 개입이 10회 초과 (단, 표준 응답 #4는 카운트 제외).

종료 사유를 notes.md에 기록.

## 6. 기록 필수 메타데이터

`runs/<harness>/notes.md` 상단에 박제:

- 모델 이름·버전·날짜
- 세션 시작/종료 시각
- 총 사람 개입 횟수 (질문 답변 / 코드 수정 분리)
- 종료 사유
- 산출 파일 위치

## 7. 채점은 사후 분리

- run 진행 중에는 `spec/notes.md`(A1~A20)를 **보지 않는다**. 무의식 편향 방지.
- 회차 종료 후에만 정답지를 열어 채점하고 `matrix.md`에 한 줄 추가.

## 8. 동일 하니스 재실행

같은 하니스 재실행은 `runs/<harness>/v2/`처럼 하위 폴더로 누적. 이전 v1 결과는 보지 않는 새 세션에서 진행.
