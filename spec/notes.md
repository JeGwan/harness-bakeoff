# Spec Notes — 사람용 (하니스에는 주지 않음)

`prompt.md`에 의도적으로 심어둔 **모호 지점**과 **채점 기준**.
이걸 보고 하니스별 산출물이 모호 지점을 얼마나 잡아내는지(혹은 그냥 한쪽으로 결정해 버리는지)를 거증한다.

## 채점 기준 (matrix.md 컬럼 정의)

- **Ambiguities surfaced** — 위 A1~A20 중 산출물(plan/spec/design/질문)에서 명시적으로 다룬 개수.
- **Decisions delegated to human** — 하니스가 사람에게 물은 결정의 수.
- **Decisions silently made** — 묻지 않고 자기가 결정한 수 (낮을수록 SDD 친화적).
- **Artifacts produced** — plan.md / requirements.md / design.md / tasks.md 등 산출 문서 종류.
- **Files in src/** — 결과 코드 파일 수.
- **LOC in src/** — 결과 코드 라인 수 (대략).
- **First runnable time** — 실제 동작할 때까지 걸린 분.
- **Human interventions** — 사람이 개입한 횟수 (질문 답 제외, 코드 수정·재실행 등).
- **Tests auto-generated** — 자동 생성된 테스트 파일 수.
- **Architecture note** — UI 렌더링 방식(SVG/Canvas/DOM), 저장 형식, 의존성 재계산 알고리즘 같은 핵심 선택을 한 줄 요약.

## 비교 핵심 가설

- **Kiro / BMAD**는 모호 지점을 EARS 또는 PM 에이전트 단계에서 사전 disambiguation할 것.
- **Claude Code Plan Mode**는 "결정해주세요" 질문 묶음으로 사용자에게 던질 것.
- **Superpower**는 skill 폴더 컨벤션에 따라 산출물의 일관성이 가장 높을 가능성.
- **Ouroboros**는 첫 회차에 모호 지점을 잡아내기보다, 반복하면서 자기 스캐폴딩 자체를 수정하는 동선이 관찰될 것.
